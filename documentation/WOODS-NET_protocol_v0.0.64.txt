I. LAYER 2/3: HARDWARE CONTROL & "FORCE WAKE" (The Scream)
The Base Station can physically boot a powered-off Mule by broadcasting a Remote AT Command to toggle a GPIO pin wired to the Pi's GLOBAL_EN/RUN pin.

MULE STANDBY CONFIG (Stored in XBee NV-RAM via ATWR):

ATBR: 0 (10 kbps - Required for wake-up lane)
ATID: 1010 (Dedicated 10k Control ID)
ATSM: 4 (Cyclic Sleep)
ATSP: 2EE0 (120 Second Sleep Period)
ATST: 7D0 (2 Second Wake/Listen Window)
Identity: ATNI string (e.g., "MULE05"). Pi parses this for Src_ID.

BASE STATION "SCREAM" PROCEDURE (DUAL-LANE RECOVERY):

Logic: The Base Station operates on a Dual-Radio Architecture.
Radio A (Command): Permanently locked to 10kbps (ATBR 0). Handles all handshakes, scheduling, and power negotiation.
Radio B (Data): Permanently locked to 200kbps (ATBR 1/Custom). Handles all Blast Data ingress.
Mode: Unicast Remote AT Command (DIO0 High) targeting specific 64-bit MAC.
Pattern: 5s Burst / 500ms Listen-Gap.
Effect: Successful Scream triggers Hardware Reset. Any active session is immediately invalidated.

VI. BIT-PACKING & ENCODING DEFINITIONS
HEARTBEAT (TYPE 6) ENCODING Byte 0: Mule Voltage (Value / 50 = Volts). Byte 1: Camera Status (Bit-Packed): [Reset_Flag (1 bit)] | [Error_Flag (1 bit)] | [Battery_Lvl (6 bits)]

Bit 7 (Reset): 1 = Fresh Boot/Scream Recovery. Base MUST clear session.
Bit 6 (Error): 1 = Hardware/Camera Module Error.
Bits 0-5 (Batt): 0-63 Integer. Scale by x1.58 for %. (Max 99.54%).

SCHEDULE ENCODING (ACTION 0x05)
Format: 32-bit Unsigned Integer (Unix Epoch timestamp).
Example: 1740801600.
The Mule persists this timestamp and calculates a sleep duration (seconds = target_epoch - current_epoch).
This duration is programmed into the DS3231 RTC as an Alarm (register 0x07). 
The ATtiny84 relies *exclusively* on the RTC's INT pin pulling LOW to physically wake up the Mule.

VII. AUTO-NEGOTIATION & UNIFIED 95-BYTE MTU
Logic: Packet size is FIXED at 95 Bytes for ALL Rungs (Except High-Speed Blast). This allows "The Trip" (Section VII.3) to occur seamlessly without altering sequence numbers or file offsets during standard transfers.

THE LADDER 
RUNG 1 (Default): 80 kbps / 40 mW (ATBR 1, ATPL 2)
IF SSACK Received AND RSSI <= [Warn]: LOCK RUNG 1.
IF No Response OR RSSI > [Warn]: Escalation Triggered -> GOTO RUNG 2.

RUNG 2: 80 kbps / 250 mW (ATBR 1, ATPL 4)
IF SSACK Received AND RSSI <= [Crit]: LOCK RUNG 2.
THE TRIP (Downshift) Trigger: See Section VII.3.

RUNG 3: 10 kbps / 250 mW (ATBR 0, ATPL 4)
IF SSACK Received: LOCK RUNG 3.
IF No Response: Retry 3 times, then Log Sys Alert (Type 13) and Sleep.

THE TRIP (SEAMLESS DOWNSHIFT)
Trigger (Any of the following): 
a. Max Retries (Same Seq) > [Trip Retries] (Default 3). 
b. Window Error: > [Trip Error Cnt] (88) in last [Trip Window] (250). 
c. RSSI Dip: >3dB below [Crit].
Action: Mule switches XBee to 10kbps (ATBR 0).
Continuity: Mule sends NEXT Sequence # at 10kbps. Base receives on Radio A.
NO Handshake or Restart required.
NO UPSHIFT: Once tripped, Mule stays at 10kbps until next sleep cycle.

VIII. RESOURCE MANAGEMENT & EXCLUSIVE LOCKING
SINGLE-THREADED INGRESS (EXCLUSIVE LOCK):
The Base Station can only accept ONE file transfer at a time.
Upon receiving Type 4 (Req Xfer), Base checks if a reservation exists.
Base validates [Pkts] matches ceil([Size] / 95) [or 235 for Blast Mode].
IF (Active_Lock == None): Base creates .tmp file, locks session to Src_ID, sends Type 5 (Approve).
IF (Active_Lock != None): Base sends Type 14 (Busy). Requesting Mule MUST Nap and retry later.

ZOMBIE WATCHDOG & STATELESS REJECTION:
Base maintains a 60s inactivity timer on the Active Lock.
If Timer Expires: Base deletes .tmp file, clears Lock, and purges ID.
REJECTION: If Base receives Type 10 (Data) from ANY Src_ID that does not match the current Active_Lock, Base MUST reply with Type 12 (Xfer Fail).

CLOUD-VERIFIED DELETE:
Base resolves names synchronously via API.
Action 0x87 (Delete) is ONLY served to Mule after Cloud Server returns 200 OK.
INTEGRITY CHECK: Base MUST calculate CRC-16 of received file and match against Type 4/15 Header CRC before authorizing upload or delete.

IX. SHUTDOWN "PRIME" ROUTINE
To ensure the hardware is ready for a future "Force Wake," the Pi MUST execute:
ATBR 0 (Set to 10k lane) - CRITICAL: Must revert regardless of Rung used.
ATID 1010 (Set to Control ID)
ATSP 2EE0 / ATST 7D0 (Configure Cyclic Sleep)
ATWR (Commit to Flash)
OS Shutdown

X. SESSION STATE MACHINE (THE "TASK LOOP")
The Session operates on a Pull-Based Model. The Mule drives the cycle.

STATE 1: INITIALIZATION
Mule Wakes -> Negotiates Ladder -> Sends Heartbeat (Type 6).
IF Base Replies (SSACK) -> GOTO STATE 2.

STATE 2: TASK SOLICITATION
Mule sends Type 7 (Task Query).
Base checks Queue for Src_ID.
IF Task Exists -> Send Type 8 (Task Assign). GOTO STATE 3.
IF Queue Empty -> Send Type 8 (Action 0x09 No Task). GOTO STATE 4.

STATE 3: EXECUTION & REPORTING
Mule attempts Task.
Mule sends Type 9 (Task Response) with Result Code.
GOTO STATE 2 (Ask for next task).

STATE 4: YIELD & UPLOAD DECISION
Mule receives Action 0x09 (No Task).
DECISION: 
A. IF Files Pending: Send Type 4 (Req Xfer).
   Upon Completion (Type 11/12): GOTO STATE 2 (Check for new tasks). 
B. IF No Files: Execute Shutdown Prime Routine. SLEEP.

XI. ASYNCHRONOUS BLAST MODE (200kbps Bypass Lane)
[NEW: Formally Ratified 2026-02-28]

To achieve 10+ KB/s throughput on perfectly clean links, the Mule is capable of disabling the hardware-level Unicast 0x8B Transmit Status ACKs natively generated by the 900HP's API parser (Frame_ID: 0) and replacing them with a purely asynchronous Windowed-Burst mechanism over SPI.

PAYLOAD GEOMETRY (The 235-Byte Core)
Because the 900HP DigiMesh firmwares natively support ~255 Byte RF frames, the Mule chunks all files into rigid 235-byte structural payloads to fully max out the hardware MTU limit.
The formal 2-byte Global Envelope is physically stripped off to prevent wasted radio airtime and bypass Base Station parser loops.
Explicit MAC Addressing Requirement: To prevent 0x25 Route Not Found dynamic routing failures during long blasts, the Mule must hardcode the API Destination 16-bit address to 0xFFFE, forcing absolute 64-bit hardware MAC routing.

THE "A-SPECTRUM" DYNAMIC WINDOW SHIFT BYPASS
Every 235-byte block is prefixed by mathematical 1-byte Control Flags: 
0xA0: 128-Packet Burst 
0xA1: 64-Packet Burst 
0xA2: 32-Packet Burst 
0xA3: 16-Packet Burst
Total Over The Air Payload: [Bypass Flag (1 byte)][Seq(2)][Data(235)] = 238 Bytes

The Base Station intercepts 0xA0-A3 instantly, routes it directly to memory, and adjusts its own modulo window math (e.g. Seq % 64 == 0) dynamically.
Dynamic Session Coupling: Because the Bypass Header omits the Src_ID, the Base Station explicitly assigns the Origin (Src_ID = 0x0) and resolves data dynamically against the active single-threaded `Session Manager Lock`.

ALGORITHMIC FLOW CONTROL PACING (Empirical Pi Zero SPI Limit)
Because SPI operates at 3 MHz and the Antenna drains at 200 Kbps, the XBee's internal buffer will instantly overflow during a window.
Benchmarking established a hard minimum 14ms API Frame processing delay inside the XBee regardless of payload size.
A synthetic locking delay is implemented on the Pi's Master SPI thread using a piecewise curve: 14ms base + ((size - 200) * 0.0002). At a 235-byte chunk, this evaluates to exactly 21 milliseconds of micro-pacing, achieving a stable ~9.96 KB/s continuous mathematical RF injection ceiling.

ERROR RECOVERY
Upon timing out (Missing Base SSACK), the Mule instantly rewinds its integer sequence pointer to the start of the current window and dynamically drops the window size (e.g. shifts from 0xA0 to 0xA1), attempting resilient progress.

XII. ITERATIVE POWER SCALING (Phase 10 & 17)
To maximize remote node battery life while remaining resilient to dynamic environmental attenuation (e.g., foliage, rain), the RF mesh employs two asymmetrical Power Level (PL) feedback loops:

1. INITIAL DYNAMIC HANDSHAKE MIRRORING (Phase 10)
Instead of statically blasting `REQ_XFER` payloads, the Mule executes an Iterative Handshake.
- The Mule begins transmitting `REQ_XFER` at `PL=0` (Lowest Power).
- If no `APPROVE` is received, it steps up: `PL=1`, `PL=2`, up to `PL=4`.
- When the Base Station receives the request, it extracts the Mule's successful `PL` integer from the packet payload. 
- *Crucially*, the Base Station dynamically modifies its *own* local XBee parameters to strictly mirror the Mule's successful `PL` index *before* returning the `APPROVE` target MAC over the mesh.

2. MID-STREAM BURST ESCALATION (Phase 17)
During an active Phase 11 Blast transmission, sudden structural interference can cause the Base Station arrays to blind.
- The Mule's sliding window specifically tracks `consecutive_failures` (dropped `SSACK` synchronizations or Swiss-Cheese patching timeouts).
- If `consecutive_failures == 3` (two attempts prior to the fatal loop abort threshold at 5), the Mule's Python execution halts.
- By issuing an immediate local AT Command `['PL', bytes([pl+1])]`, the Mule seamlessly punches the physical Radio's transmit power up by +5 dBm (e.g., `PL=0` to `PL=1`).
- The file loop instantly resumes at the new power threshold, completely bypassing the need to restart or renegotiate the master handshake.
